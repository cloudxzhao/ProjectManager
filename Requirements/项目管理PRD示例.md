## 1. 引言

### 1.1 产品背景
随着团队协作日益复杂，中小型团队需要一个轻量、易用且可定制的项目管理工具，以提升任务追踪效率、透明度和团队协作能力。当前市场上虽有众多成熟产品，但大多功能臃肿或收费昂贵，无法完全满足国内团队的个性化需求。因此，我们计划自研一款聚焦核心项目管理功能的 Web 应用，第一期实现最小可行产品（MVP）。

### 1.2 产品目标
第一期目标：打造一个简洁、高效的项目管理基础平台，支持项目与任务的创建、分配、状态跟踪，让团队成员能快速上手，实现对工作进度的可视化管控。  
核心指标：
- 项目创建与任务拆解功能完整可用
- 任务状态流转清晰
- 用户可独立完成项目与任务的日常管理
- 系统响应时间在可接受范围内，保证基础可用性

### 1.3 产品范围
第一期包含以下主要模块：
- 用户认证与权限管理（基础角色）
- 项目管理（增删改查）
- 任务管理（增删改查、状态流转、人员分配）
- 简单的个人工作台（任务概览）

第一期不包含：
- 复杂的报表统计
- 甘特图、日历视图
- 文件上传与管理
- 消息通知与邮件提醒
- 多级权限细分（如仅项目经理可删除任务）
- API 开放接口

## 2. 用户角色

| 角色     | 描述                                                        |
| -------- | ----------------------------------------------------------- |
| 管理员   | 拥有全部权限，可创建/删除项目，管理所有用户，分配项目成员。 |
| 普通成员 | 可参与项目，创建任务、编辑自己负责的任务、更新任务状态。    |

*注：第一期暂不细分项目经理角色，项目经理权限由管理员在项目中手动赋予（如设置为项目负责人）。*

## 3. 功能需求

### 3.1 用户认证与权限管理
- **注册/登录**  
  - 支持邮箱/密码注册、登录。  
  - 注册时需填写用户名、邮箱、密码。  
  - 登录后保持会话，未登录用户无法访问任何项目页面。
- **用户管理**（管理员功能）  
  - 查看所有用户列表。  
  - 禁用/启用用户账号。  
  - 重置用户密码（可发送重置链接）。

### 3.2 项目管理
- **项目列表**  
  - 以卡片或列表形式展示用户参与的项目。  
  - 展示项目名称、项目负责人、截止日期、任务完成进度（可选）。  
  - 支持按项目名称搜索。
- **项目详情**  
  - 显示项目基本信息（名称、描述、开始/结束日期、负责人）。  
  - 展示项目下的任务列表（以看板或列表形式，第一期先实现列表视图）。
- **项目操作**  
  - 创建项目：填写项目名称、描述、开始日期、结束日期，选择项目成员（从平台用户中选择）。  
  - 编辑项目：修改项目信息，调整成员。  
  - 删除项目：只有管理员或项目负责人可删除，删除前需二次确认，同时删除项目下所有任务。

### 3.3 任务管理
- **任务列表**  
  - 在项目详情页内，以表格或简单列表展示所有任务。  
  - 列表列包括：任务标题、负责人、截止日期、状态、优先级（可选）。  
  - 支持按状态、负责人、截止日期筛选。
- **任务操作**  
  - **创建任务**：填写标题、描述、截止日期、负责人（从项目成员中选择）、状态（默认“待处理”）、优先级（高/中/低）。  
  - **编辑任务**：修改所有字段。  
  - **删除任务**：只有创建者、项目负责人或管理员可删除，需二次确认。  
  - **更改状态**：状态包括“待处理”、“进行中”、“已完成”。支持一键切换。  
  - **分配负责人**：可在任务详情页更改负责人。
- **任务详情页**  
  - 集中展示任务所有信息，并支持编辑。  
  - 可添加简单评论（仅文字，作为沟通记录）。  
  - 显示任务变更历史（如状态变化、负责人变更，第一期可选）。

### 3.4 个人工作台
- **我的任务**  
  - 登录后首页展示当前用户负责的所有未完成任务（待处理、进行中）。  
  - 每个任务卡片包含项目名称、任务标题、截止日期。  
  - 点击卡片跳转到对应任务详情。
- **快捷操作**  
  - 快速更改任务状态（下拉选择或按钮）。

## 4. 非功能需求

### 4.1 性能要求
- 页面加载时间不超过 2 秒（在正常网络环境下）。
- 支持并发用户数不少于 50 人（初期规模）。
- 任务列表分页加载，每页默认 20 条。

### 4.2 安全性要求
- 用户密码加密存储，不可明文保存。
- 所有请求需校验用户登录状态，防止越权访问。
- 前后端交互使用 HTTPS（如部署时配置）。
- 防止 XSS、CSRF 等常见 Web 攻击。

### 4.3 可用性要求
- 界面简洁直观，符合主流项目管理工具的操作习惯。
- 提供必要的操作引导（如空状态提示、首次使用引导）。
- 移动端适配：至少保证核心功能在手机浏览器上可操作（响应式设计）。

## 5. 优先级划分（MoSCoW 方法）

| 优先级      | 需求描述                                                     |
| ----------- | ------------------------------------------------------------ |
| Must have   | 用户注册登录、项目创建与编辑、任务创建与编辑、任务状态变更、任务分配、个人工作台 |
| Should have | 项目成员管理、任务筛选与搜索、任务评论、任务变更历史         |
| Could have  | 项目进度统计（简单百分比）、任务优先级、项目归档             |
| Won't have  | 文件上传、甘特图、第三方集成、消息通知、API                  |

## 6. 附录

### 6.1 数据模型（简略）
- **User**：id, username, email, password_hash, is_active, is_admin, created_at
- **Project**：id, name, description, start_date, end_date, owner_id (FK User), created_at
- **ProjectMember**：id, project_id (FK Project), user_id (FK User), role (可选，预留)
- **Task**：id, title, description, status (enum: todo, doing, done), priority (enum: high, medium, low), due_date, project_id (FK Project), assignee_id (FK User), creator_id (FK User), created_at, updated_at
- **Comment**：id, content, task_id (FK Task), user_id (FK User), created_at

### 6.2 术语表
- **项目负责人**：创建项目时指定的管理员，拥有项目内最高权限。
- **任务状态**：待处理、进行中、已完成。
- **优先级**：高、中、低。

---

## 7. 第一期验收标准
1. 用户可完成注册、登录，并正确跳转至个人工作台。
2. 管理员可创建项目，添加成员；成员可查看自己参与的项目。
3. 在项目内可创建任务，并分配给项目成员。
4. 任务状态可自由切换，且所有成员可见。
5. 个人工作台准确展示当前用户负责的任务。
6. 系统无明显性能瓶颈，核心流程无阻断性 Bug。
7. 所有界面风格统一，操作反馈明确。
