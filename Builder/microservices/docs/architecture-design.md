# ProjectHub 微服务架构设计文档

## 1. 架构概述

### 1.1 架构愿景

ProjectHub 微服务架构旨在将单体应用拆分为多个独立部署、独立扩展的微服务，以提高系统的可扩展性、可维护性和开发效率。

### 1.2 架构原则

- **服务自治**: 每个服务拥有独立的数据库，可独立部署和扩展
- **领域驱动**: 按业务领域拆分服务，保持高内聚低耦合
- **API 优先**: 通过 API Gateway 统一对外暴露接口
- **事件驱动**: 服务间通过事件进行异步通信
- **可观测性**: 完善的日志、指标、链路追踪

## 4. 核心设计

### 4.1 认证授权设计

```
┌──────────┐     ┌─────────────┐     ┌──────────────┐
│  Client  │────▶│   Gateway   │────▶│ Auth Service │
└──────────┘     └─────────────┘     └──────────────┘
     │                  │                    │
     │  1. 登录请求      │                    │
     ├─────────────────>│                    │
     │                  │  2. 转发           │
     │                  ├───────────────────>│
     │                  │                    │
     │                  │  3. 生成 Token      │
     │                  │<───────────────────│
     │  4. 返回 Token    │                    │
     │<─────────────────│                    │
     │                  │                    │
     │  5. 业务请求      │                    │
     │  (携带 Token)     │                    │
     ├─────────────────>│                    │
     │                  │  6. 验证 Token      │
     │                  │     注入用户信息     │
     │                  │                    │
     │                  │  7. 转发 + Header   │
     │                  ├───────────────────>│
     │                  │                    │
     │                  │  8. 业务处理        │
     │                  │<───────────────────│
     │  9. 返回结果      │                    │
     │<─────────────────│                    │
```

**JWT Token 结构**:
```json
{
  "sub": "123",
  "username": "admin",
  "role": "ADMIN",
  "iat": 1710144000,
  "exp": 1710230400
}
```

**Gateway 注入 Header**:
- `X-User-Id`: 用户 ID
- `X-User-Name`: 用户名
- `X-User-Role`: 用户角色
- `Authorization`: JWT Token

### 4.2 服务间通信

#### 同步调用 (REST/Feign)

```java
@FeignClient(name = "user-service")
public interface UserClient {
    @GetMapping("/api/v1/users/internal/{id}")
    Result<UserVO> getUserById(@PathVariable Long id);
}
```

#### 异步通信 (RabbitMQ)

```
┌──────────────┐    ┌─────────────┐    ┌──────────────┐
│ Task Service │───▶│  RabbitMQ   │───▶│ Notification │
│              │    │   Exchange  │    │   Service    │
└──────────────┘    └─────────────┘    └──────────────┘
     发布事件              路由                消费事件
```

### 4.3 数据库设计

#### 数据库拆分

```
单体数据库 (projectdb)
├── users
├── projects
├── tasks
└── ...

拆分为:

auth_db         ──→ users, refresh_tokens
user_db         ──→ users, user_profiles
project_db      ──→ projects, project_members
task_db         ──→ tasks, comments
story_db        ──→ epics, user_stories
issue_db        ──→ issues
wiki_db         ──→ wiki_spaces, wiki_pages
notify_db       ──→ notifications
```

#### 数据冗余策略

```
user_db (主数据)
  │
  │ 用户信息变更事件
  │ (RabbitMQ 广播)
  ▼

auth_db ──→ 基本信息冗余 (username, email)
project_db ──→ 基本信息冗余 (username, avatar)
task_db ──→ 基本信息冗余 (assignee_name)
```

### 4.4 分布式事务

#### AT 模式 (推荐)

```java
@GlobalTransactional
public void deleteProjectWithTasks(Long projectId) {
    // 1. 删除项目
    projectRepository.deleteById(projectId);

    // 2. 删除项目成员
    projectMemberRepository.deleteByProjectId(projectId);

    // 3. 删除任务（远程调用）
    taskClient.deleteByProjectId(projectId);

    // 任何一步失败，全局回滚
}
```

#### 最终一致性 (事件驱动)

```
服务 A ──→ 本地事务 ──→ 发布事件 ──→ RabbitMQ
                                          │
                                          ▼
服务 B ←── 本地事务 ←── 消费事件 ←───────┘
```

## 7. 安全设计

### 7.1 认证机制

- **JWT Token**: 无状态认证，Gateway 统一验证
- **Token 刷新**: Refresh Token 机制，延长用户会话
- **服务间认证**: Service-to-Service Token

### 7.2 授权机制

- **RBAC**: 基于角色的访问控制
- **API 权限**: 细粒度到接口级别
- **数据权限**: 项目级别数据隔离

### 7.3 数据安全

- **SQL 注入**: MyBatis-Plus 参数化查询
- **XSS 防护**: 输入验证 + 输出转义
- **敏感数据**: 密码 BCrypt 加密存储

## 9. 决策记录

### D1: 为什么选择 Nacos 而不是 Eureka?

**决策**: 选择 Nacos

**理由**:
- 同时支持服务发现和配置中心
- 支持 AP 和 CP 模式
- 阿里生态，与 Spring Cloud Alibaba 集成好
- 中文文档和社区支持

### D2: 为什么选择 RabbitMQ 而不是 Kafka?

**决策**: 选择 RabbitMQ

**理由**:
- 项目规模中等，RabbitMQ 足够
- 支持多种路由模式
- 支持延迟队列（通知场景）
- 运维简单，学习曲线低

### D3: 为什么选择 AT 模式而不是 TCC?

**决策**: 选择 Seata AT 模式

**理由**:
- 代码侵入性低
- 开发成本低
- 性能满足业务需求
- TCC 需要实现三个接口，过于复杂
