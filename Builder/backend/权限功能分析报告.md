# ProjectHub 权限功能分析报告

## 一、当前权限架构概述

### 1.1 权限模型

当前项目采用 **基于角色的访问控制 (RBAC)** 与 **项目级权限** 相结合的混合模型：

```
┌─────────────────────────────────────────────────────────────┐
│                    权限架构层次                              │
├─────────────────────────────────────────────────────────────┤
│  第一层：全局权限 (Global Permission)                        │
│  - 基于用户角色 (ROLE_USER, ROLE_ADMIN 等)                   │
│  - 通过 Spring Security 的 GrantedAuthority 管理              │
│                                                             │
│  第二层：项目级权限 (Project-Level Permission)               │
│  - 项目所有者 (Owner) - 项目创建者                           │
│  - 项目管理员 (Admin)                                        │
│  - 项目经理 (Manager)                                        │
│  - 普通成员 (Member)                                         │
│                                                             │
│  第三层：功能权限 (Function Permission)                      │
│  - TASK_CREATE, TASK_EDIT, TASK_DELETE 等                   │
│  - WIKI_CREATE, WIKI_EDIT, STORY_VIEW 等                    │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心组件

| 组件 | 路径 | 职责 |
|------|------|------|
| `PermissionService` | `module/project/service/` | 权限校验核心逻辑 |
| `PermissionAspect` | `common/aspect/` | AOP 切面拦截校验 |
| `RequirePermission` | `common/aspect/` | 权限注解定义 |
| `ProjectMember` | `module/project/entity/` | 项目成员关系与角色 |
| `UserDetailsImpl` | `security/` | Spring Security 用户详情 |

## 三、权限设计合理性分析

### ✅ 3.1 优点

| 优点 | 说明 |
|------|------|
| **分层清晰** | 全局权限 + 项目权限 + 功能权限三层设计 |
| **AOP 解耦** | 使用切面进行权限校验，业务代码解耦 |
| **注解驱动** | `@RequirePermission` 注解标记，声明式权限控制 |
| **角色枚举** | `ProjectMemberRole` 枚举定义清晰 (OWNER/ADMIN/MANAGER/MEMBER) |
| **统一校验** | `PermissionService` 统一入口，便于维护 |
| **Owner 特殊处理** | 项目创建者自动拥有最高权限 |

### ⚠️ 3.2 存在的问题

#### 问题 1: 权限校验逻辑不完整

**位置**: `PermissionService.hasPermission()` (第 19-33 行)

```java
public boolean hasPermission(Long userId, Long projectId, String permissionCode) {
    ProjectMember member = projectMemberRepository
        .findByProjectIdAndUserId(projectId, userId)
        .orElse(null);

    if (member == null) {
        // 检查是否是项目所有者
        return projectRepository.existsByIdAndOwnerId(projectId, userId);
    }

    // ❌ 问题：项目成员默认有基本权限，没有根据 permissionCode 细化校验
    return true;  // ← 这里直接返回 true，权限代码未生效
}
```

**影响**:
- 所有项目成员默认拥有所有权限
- `TASK_DELETE`、`TASK_MOVE` 等高级权限无法限制
- 权限代码 `permissionCode` 参数形同虚设

---

#### 问题 2: 全局权限未实现

**位置**: `PermissionService.hasGlobalPermission()` (第 35-41 行)

```java

    // TODO: 实现全局权限检查（基于角色和权限表）

    return true;
```

**影响**:
- 管理员角色权限未定义

- TODO 代码遗留

#### 问题 3: 角色 - 权限映射缺失

当前定义了 4 个角色：
```java
public enum ProjectMemberRole {
    OWNER,    // 项目所有者
    ADMIN,    // 项目管理员
    MANAGER,  // 项目经理
    MEMBER    // 普通成员
}
```

但**没有定义每个角色的具体权限范围**：

| 角色 | TASK_CREATE | TASK_EDIT | TASK_DELETE | TASK_MOVE | 成员管理 |
|------|-------------|-----------|-------------|-----------|----------|
| OWNER | ✅ | ✅ | ✅ | ✅ | ✅ |
| ADMIN | ✅ | ✅ | ✅ | ✅ | ✅ |
| MANAGER | ✅ | ✅ | ❓ | ✅ | ❓ |
| MEMBER | ✅ | ❓ | ❓ | ❓ | ❓ |

**问题**:
- 角色权限映射表缺失（靠硬编码而非配置）
- MANAGER 和 MEMBER 的权限边界模糊

---

#### 问题 4: 权限校验不一致

**问题代码对比**:

```java
// TaskService - 手动校验
if (!permissionService.hasPermission(userId, projectId, "TASK_CREATE")) {
    throw new BusinessException(ErrorCode.TASK_PERMISSION_DENIED, "无创建任务权限");
}

// PermissionAspect - AOP 自动校验
if (projectId != null && !permissionService.hasPermission(userId, projectId, permissionCode)) {
    throw new BusinessException(403, "权限不足");
}
```

**问题**:
1. 部分 Service 手动校验，部分依赖 AOP
2. 异常信息不统一（有的带错误码，有的直接用 403）
3. AOP 切面在 `PermissionAspect` 第 56 行全局权限校验逻辑有问题：

```java
// 如果 projectId == null，会检查全局权限
if (!permissionService.hasGlobalPermission(userId, permissionCode) && projectId == null) {
    // ❌ 逻辑问题：hasGlobalPermission 返回 true 时会跳过
}
```

#### 问题 5: 数据库表设计缺失

**问题**: 没有独立的权限表

当前数据库结构：
```
project_member
├── id
├── project_id
├── user_id
└── role (OWNER/ADMIN/MANAGER/MEMBER)
```

**缺失**:
- `role_permission` 表（角色 - 权限映射）
- `user_role` 表（用户 - 全局角色）
- 权限配置表

#### 问题 6: 权限缓存缺失

**问题**: 每次请求都查询数据库

```java
@Transactional(readOnly = true)
public boolean hasPermission(Long userId, Long projectId, String permissionCode) {
    // 每次都查数据库
    ProjectMember member = projectMemberRepository
        .findByProjectIdAndUserId(projectId, userId)
        .orElse(null);
    // ...
}
```

**影响**:
- 高频查询场景性能问题（如列表接口批量校验）
- 未使用 Redis 缓存用户权限

---

#### 问题 7: 跨项目权限未处理

**场景**: 用户同时属于多个项目


**问题**:
- 跨项目资源访问控制复杂
- 全局任务搜索可能泄露无权限项目的数据

## 四、改进建议

### 4.1 短期改进（P1 优先级）

#### 建议 1: 实现角色 - 权限映射

```java
@Service
public class PermissionService {

    // 角色权限映射表
    private static final Map<ProjectMemberRole, Set<String>> ROLE_PERMISSIONS = Map.of(
        ProjectMemberRole.OWNER, Set.of("*"),  // 所有权限
        ProjectMemberRole.ADMIN, Set.of(
            "TASK_CREATE", "TASK_EDIT", "TASK_DELETE", "TASK_MOVE",
            "STORY_CREATE", "STORY_EDIT", "STORY_DELETE",
            "EPIC_CREATE", "EPIC_EDIT", "EPIC_DELETE",
            "ISSUE_CREATE", "ISSUE_EDIT", "ISSUE_DELETE",
            "WIKI_CREATE", "WIKI_EDIT", "WIKI_DELETE",
            "MEMBER_MANAGE"
        ),
        ProjectMemberRole.MANAGER, Set.of(
            "TASK_CREATE", "TASK_EDIT", "TASK_MOVE",
            "STORY_CREATE", "STORY_EDIT",
            "EPIC_CREATE", "EPIC_EDIT",
            "ISSUE_CREATE", "ISSUE_EDIT",
            "WIKI_CREATE", "WIKI_EDIT"
        ),
        ProjectMemberRole.MEMBER, Set.of(
            "TASK_CREATE", "TASK_EDIT",  // 自己创建的任务
            "TASK_COMMENT",
            "STORY_VIEW", "EPIC_VIEW", "ISSUE_VIEW", "WIKI_VIEW"
        )
    );

    public boolean hasPermission(Long userId, Long projectId, String permissionCode) {
        ProjectMember member = projectMemberRepository
            .findByProjectIdAndUserId(projectId, userId)
            .orElse(null);

        if (member == null) {
            return projectRepository.existsByIdAndOwnerId(projectId, userId);
        }

        // 根据角色和权限码校验
        return hasRolePermission(member.getRole(), permissionCode);
    }

    private boolean hasRolePermission(ProjectMemberRole role, String permissionCode) {
        Set<String> permissions = ROLE_PERMISSIONS.get(role);
        return permissions != null && (permissions.contains("*") || permissions.contains(permissionCode));
    }
}
```

#### 建议 2: 修复 AOP 逻辑

```java
@Around("@annotation(com.projecthub.common.aspect.RequirePermission)")
public Object checkPermission(ProceedingJoinPoint joinPoint) throws Throwable {
    MethodSignature signature = (MethodSignature) joinPoint.getSignature();
    Method method = signature.getMethod();
    RequirePermission requirePermission = method.getAnnotation(RequirePermission.class);

    // 获取当前用户
    Object principal = SecurityContextHolder.getContext().getAuthentication().getPrincipal();
    if (!(principal instanceof UserDetailsImpl userDetails)) {
        throw new BusinessException(ErrorCode.UNAUTHORIZED, "用户未登录");
    }

    Long userId = userDetails.getId();
    String permissionCode = requirePermission.value();
    Long projectId = extractProjectId(joinPoint, requirePermission.projectIdParam());

    // 有项目 ID 则检查项目权限，否则检查全局权限
    if (projectId != null) {
        if (!permissionService.hasPermission(userId, projectId, permissionCode)) {
            throw new BusinessException(ErrorCode.PROJECT_PERMISSION_DENIED, "权限不足");
        }
    } else if (!permissionService.hasGlobalPermission(userId, permissionCode)) {
        throw new BusinessException(ErrorCode.FORBIDDEN, "权限不足");
    }

    return joinPoint.proceed();
}
```

#### 建议 3: 添加权限缓存

```java
@Service
@RequiredArgsConstructor
public class PermissionService {

    private final RedisTemplate<String, Object> redisTemplate;

    // 缓存键前缀
    private static final String PERMISSION_CACHE_KEY = "permission:user:";

    public boolean hasPermission(Long userId, Long projectId, String permissionCode) {
        String cacheKey = PERMISSION_CACHE_KEY + userId + ":project:" + projectId;

        // 从缓存获取
        Set<String> cachedPermissions = (Set<String>) redisTemplate.opsForValue().get(cacheKey);
        if (cachedPermissions != null) {
            return cachedPermissions.contains(permissionCode) || cachedPermissions.contains("*");
        }

        // 数据库查询
        ProjectMember member = projectMemberRepository
            .findByProjectIdAndUserId(projectId, userId)
            .orElse(null);

        if (member == null) {
            boolean isOwner = projectRepository.existsByIdAndOwnerId(projectId, userId);
            return isOwner;
        }

        Set<String> permissions = getPermissionsByRole(member.getRole());

        // 写入缓存（5 分钟）
        redisTemplate.opsForValue().set(cacheKey, permissions, 5, TimeUnit.MINUTES);

        return permissions.contains(permissionCode) || permissions.contains("*");
    }

    // 缓存失效：成员变更时
    @CacheEvict(key = "'permission:user:' + #userId + ':project:' + #projectId")
    public void evictPermissionCache(Long userId, Long projectId) {
        // Spring Cache 自动处理
    }
}
```

### 4.2 中期改进（P2 优先级）

#### 建议 4: 数据库表扩展

```sql
-- 全局角色表
CREATE TABLE sys_role (
    id BIGSERIAL PRIMARY KEY,
    role_code VARCHAR(50) UNIQUE NOT NULL,  -- ROLE_ADMIN, ROLE_USER
    role_name VARCHAR(100) NOT NULL,
    description TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 全局权限表
CREATE TABLE sys_permission (
    id BIGSERIAL PRIMARY KEY,
    permission_code VARCHAR(100) UNIQUE NOT NULL,  -- USER_MANAGE, PROJECT_MANAGE
    permission_name VARCHAR(100) NOT NULL,
    module VARCHAR(50) NOT NULL,  -- 所属模块
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 角色 - 权限关联表
CREATE TABLE role_permission (
    id BIGSERIAL PRIMARY KEY,
    role_id BIGINT NOT NULL REFERENCES sys_role(id),
    permission_id BIGINT NOT NULL REFERENCES sys_permission(id),
    UNIQUE(role_id, permission_id)
);

-- 用户 - 角色关联表
CREATE TABLE user_role (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL REFERENCES user(id),
    role_id BIGINT NOT NULL REFERENCES sys_role(id),
    UNIQUE(user_id, role_id)
);

-- 项目角色权限表（项目级特殊配置）
CREATE TABLE project_role_permission (
    id BIGSERIAL PRIMARY KEY,
    project_id BIGINT NOT NULL REFERENCES project(id),
    role_id BIGINT NOT NULL REFERENCES sys_role(id),
    permission_code VARCHAR(100) NOT NULL,
    UNIQUE(project_id, role_id, permission_code)
);
```

```java
@Aspect
@Component
public class PermissionAspect {

    @Around("@annotation(com.projecthub.common.aspect.RequirePermission)")
    public Object checkPermission(ProceedingJoinPoint joinPoint) throws Throwable {
        try {
            // ... 权限校验逻辑

            log.warn("权限校验失败：userId={}, permission={}, error={}",

            log.error("权限校验异常", e);

    }
```

## 五、总结

### 5.1 当前状态评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | ⭐⭐⭐⭐ | 分层清晰，AOP 解耦 |
| 功能完整 | ⭐⭐ | 角色权限映射缺失，TODO 代码 |

| 可扩展性 | ⭐⭐⭐ | 支持扩展但需要重构 |
| 性能 | ⭐⭐ | 无缓存，高频查询 DB |

### 5.2 优先改进清单

1. **P0**: 实现角色 - 权限映射（修复 `hasPermission` 返回 `true` 问题）
2. **P0**: 修复 AOP 全局权限校验逻辑
3. **P1**: 添加权限缓存
4. **P1**: 统一权限异常处理
5. **P2**: 扩展数据库表支持全局角色

### 5.3 风险提示

⚠️ **高风险**: 当前所有项目成员默认拥有所有权限，存在严重安全隐患，建议优先修复。
