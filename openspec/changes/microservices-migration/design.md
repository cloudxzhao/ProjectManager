### 约束条件
- 保持 API 向后兼容，前端改动最小化
- 支持渐进式迁移，不要求一次性切换
- 使用 Spring Cloud Alibaba 技术栈
- 保持现有认证机制（JWT）有效性

## Goals / Non-Goals

**Goals:**
- 将单体应用拆分为 8 个独立微服务
- 实现服务间通过 API Gateway 统一访问
- 每个服务拥有独立数据库，支持独立部署
- 实现服务发现、配置中心、链路追踪
- 实现事件驱动的异步通信机制
- 保证 99.9% 服务可用性

**Non-Goals:**
- 本阶段不引入 Kubernetes（使用 Docker Compose 部署）
- 不引入 Service Mesh（Istio/Linkerd）
- 不改变现有前端架构
- 不引入 GraphQL（保持 REST API）

## Decisions

### D1: 微服务拆分策略

**决策**: 按业务领域拆分（DDD 领域驱动设计）

```
┌─────────────────────────────────────────────────────────────────┐
│                        API Gateway (8080)                       │
└───────────────────────────────┬─────────────────────────────────┘

        ┌───────────────────────┼───────────────────────┐
        │                       │                       │
        ▼                       ▼                       ▼
┌───────────────┐     ┌───────────────┐     ┌───────────────┐
│  Auth Service │     │ User Service  │     │Project Service│
│   (9521)      │     │   (9522)      │     │   (9523)      │
└───────────────┘     └───────────────┘     └───────────────┘
        │                       │                       │
        │         ┌─────────────┼─────────────┐         │
        │         │             │             │         │
        ▼         ▼             ▼             ▼         ▼
┌───────────────┐ ┌───────────────┐ ┌───────────────┐ ┌───────────────┐
│ Task Service  │ │ Story Service │ │Issue Service  │ │Wiki Service   │
│   (9524)      │ │   (9525)      │ │   (9526)      │ │   (9527)      │
└───────────────┘ └───────────────┘ └───────────────┘ └───────────────┘
        │
        ▼
┌───────────────┐
│Notification   │
│ Service (9528)│
└───────────────┘
```

**备选方案**:
| 方案 | 优点 | 缺点 |
|------|------|------|
| 按业务领域拆分（选用） | 高内聚、低耦合、团队自治 | 初期边界划分需要仔细思考 |
| 按功能拆分 | 简单快速 | 服务数量多、服务间调用频繁 |
| 按团队拆分 | 团队独立性强 | 可能造成服务边界不清 |

**理由**: 业务领域拆分最符合 DDD 原则，长期维护成本最低。

### D2: 服务间通信

**决策**: 同步调用使用 OpenFeign，异步使用 RabbitMQ

```
同步调用（强依赖）:
┌─────────┐    Feign/REST    ┌─────────┐
│ Service │ ───────────────▶ │ Service │
│    A    │                  │    B    │
└─────────┘                  └─────────┘

异步调用（解耦）:
┌─────────┐                  ┌─────────┐
│ Service │    RabbitMQ      │ Service │
│    A    │ ──▶ [Queue] ───▶ │    B    │
└─────────┘                  └─────────┘
```

**通信矩阵**:
| 调用方 | 被调用方 | 通信方式 | 场景 |
|--------|----------|----------|------|
| Gateway | All Services | REST | 路由请求 |
| Project Service | User Service | Feign | 获取用户信息 |
| Task Service | User Service | Feign | 获取负责人信息 |
| Task Service | Project Service | Feign | 获取项目信息 |
| Task Service | Notification | RabbitMQ | 任务分配通知 |
| Auth Service | User Service | Feign | 登录验证 |
| Wiki Service | Notification | RabbitMQ | 文档更新通知 |

### D4: 认证授权方案

**决策**: Gateway 统一认证 + 本地缓存优化 + 服务间 Token 传递

```
┌──────────┐     ┌──────────────────────┐     ┌──────────┐
│  Client  │────▶│       Gateway        │────▶│ Service  │
└──────────┘     └──────────────────────┘     └──────────┘
                        │
                   ┌────┴────┐
                   │ 本地缓存 │
                   │ JWT公钥  │
                   └─────────┘
                        │
                   1. 本地验证 JWT（缓存公钥）
                   2. 解析用户信息
                   3. 注入 Header
                        │
                        ▼
                X-User-Id: 123
                X-User-Role: PM
                X-User-Name: 张三
```

**Token 流程**:
1. 用户登录 → Auth Service 返回 JWT
2. 请求携带 JWT → Gateway 本地验证（缓存 JWT 公钥，避免每次请求 Auth Service）
3. Gateway 解析并注入用户信息到 Header
4. 下游服务从 Header 获取用户信息

**缓存策略**:
- Gateway 本地缓存 JWT 公钥，TTL 1 小时
- 公钥轮换时通过 Nacos 配置推送更新
- 用户信息可选缓存到 Redis（高频访问场景）

### D5: 技术栈选型

| 组件 | 选型 | 版本 | 理由 |
|------|------|------|------|
| 注册中心 | Nacos | 2.x | 同时支持配置中心，降低运维复杂度 |
| 配置中心 | Nacos | 2.x | 与注册中心统一 |
| API Gateway | Spring Cloud Gateway | 4.x | 原生支持，性能好 |
| 服务调用 | OpenFeign | 3.x | 声明式，易用 |
| 熔断降级 | Sentinel | 1.8.x | 阿里生态，与 Nacos 集成好 |
| 消息队列 | RabbitMQ | 3.x | 成熟稳定，支持延迟队列 |
| 链路追踪 | SkyWalking | 8.x | 无侵入，可视化好，8.x 版本稳定 |
| 分布式事务 | Seata | 1.5.x | AT 模式简单易用，1.5.x 兼容性好 |

### D6: 负载均衡策略

**决策**: 默认使用加权轮询，支持按场景切换

| 策略 | 使用场景 | 说明 |
|------|----------|------|
| 加权轮询（默认） | 通用场景 | 按服务配置权重分配流量 |
| 一致性哈希 | 需要会话粘性 | 相同用户 ID 路由到同一实例 |
| 最少连接数 | 长连接场景 | 路由到当前连接数最少的实例 |

### D7: 数据同步延迟目标

| 数据类型 | 同步延迟目标 | 实现方式 |
|----------|--------------|----------|
| 用户信息变更 | < 1 秒 | RabbitMQ 广播 + 本地缓存更新 |
| 项目成员变更 | < 1 秒 | RabbitMQ 广播 |
| 事件消息投递 | < 100ms | RabbitMQ 持久化队列 |
| 配置变更 | < 5 秒 | Nacos 配置推送 |

## Risks / Trade-offs

### R1: 分布式事务复杂性
**风险**: 跨服务事务难以保证一致性
**缓解**:
- 优先使用最终一致性（事件驱动）
- 关键场景使用 Seata AT 模式
- 设计幂等接口支持重试

### R2: 运维复杂度增加
**风险**: 服务数量多，监控排查困难
**缓解**:
- 统一日志格式，接入 ELK
- 完善健康检查和告警
- 使用 SkyWalking 链路追踪

### R3: 数据迁移风险
**风险**: 现有数据需要拆分到多个数据库
**缓解**:
- 编写迁移脚本，支持回滚
- 灰度迁移，新旧并行验证
- 保留原始数据备份

### R4: 学习曲线
**风险**: 团队对微服务架构不熟悉
**缓解**:
- 提供培训和技术分享
- 编写详细的开发指南
- 代码审查确保规范

### Trade-offs

| 权衡 | 选择 | 放弃 |
|------|------|------|
| 一致性 vs 可用性 | 可用性优先（AP） | 强一致性 |
| 简单性 vs 可扩展性 | 可扩展性 | 单体简单性 |
| 性能 vs 可观测性 | 可观测性 | 少量性能损耗 |

## Open Questions

### 已关闭的决策

#### Q1: 服务间数据同步策略
**问题**: 用户信息变更后，如何快速同步到所有缓存服务？

**决策**: 采用事件广播 + 本地缓存方案
- 用户信息变更时，User Service 发布 `UserChangedEvent` 到 RabbitMQ
- 各服务监听事件并更新本地缓存
- 缓存 TTL 设置为 10 分钟（兜底策略）
- **延迟目标**: < 1 秒

#### Q2: 数据库选型
**问题**: 是否考虑部分服务使用不同数据库？

**决策**: 统一使用 PostgreSQL，后期可按需调整
- 初期统一 PostgreSQL 降低运维复杂度
- Wiki Service 后期可迁移到 MongoDB（文档存储特性）
- Auth Service 保持 PostgreSQL（事务一致性要求高）
- **当前决策**: 全部 PostgreSQL

#### Q3: 灰度发布策略
**问题**: 如何支持部分用户先使用新服务？

**决策**: 基于请求特征的灰度路由
- Gateway 支持 `X-Gray-Version` Header 灰度
- 支持用户 ID 白名单灰度
- 支持百分比流量灰度
- 详见 `api-gateway` Spec 中灰度路由需求

#### Q4: 前端改造范围
**问题**: 是否需要配合改造前端 API 调用方式？

**决策**: 保持 API 路径不变，最小化前端改动
- 所有 API 路径保持 `/api/v1/*` 格式
- 仅需修改 `NEXT_PUBLIC_API_URL` 指向 Gateway
- 无需修改前端业务代码
